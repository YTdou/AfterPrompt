# 项目任务：开发一个面向 AI 生成视觉内容的可视化编辑工具

## 一、项目背景

我经常使用 GPT、Codex 等 AI 工具生成以下视觉内容：

* SVG 图片；
* HTML/CSS 页面；
* HTML Slides 或类似 SlideV 的演示页面；
* 由 HTML、CSS、SVG 和图片素材组合而成的视觉稿。

这些生成结果通常已经有较高的完成度，但仍会存在少量需要人工微调的内容，例如：

* 修改文字内容、字体、字号或颜色；
* 移动、缩放、旋转某个元素；
* 替换图片、图标或 SVG 图形；
* 调整元素间距、对齐方式或层级关系；
* 修改某个局部布局；
* 删除或隐藏某个元素；
* 调整背景、边框、圆角、透明度等样式。

目前，非技术用户很难直接编辑这些 AI 生成的内容。

即使只需要修改一个很小的地方，用户也常常需要重新向 GPT 描述需求，让 GPT 修改甚至重新生成整个页面或整张图片。这会带来以下问题：

1. 局部修改容易影响已经满意的其他区域；
2. 用户需要反复描述、生成和筛选结果；
3. 消耗大量 Token 和交互时间；
4. 重新生成整幅内容会产生不必要的推理计算和能耗；
5. 非技术用户无法直接操作 HTML、CSS、DOM 或 SVG 源代码。

因此，我希望开发一个专门解决 AI 生成视觉内容“最后一公里编辑”问题的工具。

---

# 二、产品定位

开发一个同时面向 AI 和普通用户的视觉内容编辑工作台。

它需要同时满足两类使用方式：

## 1. 面向普通用户

用户可以像编辑 PowerPoint 一样，在画布中直接操作元素，例如：

* 点击选择；
* 拖动位置；
* 缩放尺寸；
* 旋转；
* 修改文字；
* 替换图片；
* 调整颜色和样式；
* 改变图层顺序；
* 对齐和分布元素；
* 撤销和重做。

用户不需要理解 HTML、CSS、DOM 或 SVG 代码。

## 2. 面向 GPT / Codex

所有编辑结果都应当保留为结构清晰、可继续修改的代码或结构化数据，使 GPT 或 Codex 可以：

* 读取当前页面结构；
* 精确定位具体元素；
* 修改指定元素；
* 添加或删除元素；
* 调整布局和样式；
* 批量执行修改；
* 输出修改后的 HTML、CSS 或 SVG；
* 尽量避免重新生成整个页面。

最终目标是实现一种“双向编辑”模式：

> 用户在画布中修改元素，工具自动更新底层代码；Codex 修改底层代码后，画布能够同步显示结果。

# 三、本阶段目标

本阶段只开发一个能够运行和验证核心功能的代码工具。

暂时不需要：

* 包装成精美的商业 App；
* 完整的品牌视觉设计；
* 复杂的账号系统；
* 云端协作；
* 商业化支付功能；
* 移动端适配；
* 原生桌面客户端；
* 对普通 PNG、JPEG 位图进行真正的语义分层；
* 类似 Photoshop 的像素级编辑能力；
* AI 图像局部重绘或生成式填充。

本阶段优先保证核心编辑能力、代码结构和可扩展性。

# 四、MVP 支持范围

MVP 暂时只支持以下两类内容：

## 1. HTML/CSS/DOM 内容

包括：

* 普通 HTML 页面；
* HTML Slides；
* 使用绝对定位、Flexbox 或 Grid 布局的页面；
* HTML 中的文本、图片、形状和容器；
* HTML 内嵌 SVG；
* HTML 中引用的本地图片资源。

## 2. SVG 内容

包括：

* `text`；
* `rect`；
* `circle`；
* `ellipse`；
* `line`；
* `polyline`；
* `polygon`；
* `path`；
* `image`；
* `g` 分组；
* 基本的 `transform`；
* 填充、描边、透明度等常用属性。

暂时不要求完整支持所有 SVG 标准特性，但代码结构需要为后续扩展保留空间。

# 五、核心使用流程

## 场景 A：编辑 AI 生成的 HTML Slide

1. 用户导入一个 HTML 文件或粘贴 HTML/CSS 代码；
2. 工具在画布中渲染页面；
3. 用户点击某个文字、图片、图形或容器；
4. 工具显示该元素的选中框；
5. 用户拖动、缩放、旋转或修改其属性；
6. 工具同步更新底层 HTML/CSS；
7. 用户可以查看修改后的代码；
8. 用户导出完整可运行的 HTML 文件或项目目录。

## 场景 B：编辑 AI 生成的 SVG

1. 用户导入 SVG 文件或粘贴 SVG 代码；
2. 工具在画布中渲染 SVG；
3. 用户选中某个 SVG 元素；
4. 用户移动、缩放、旋转或修改属性；
5. 工具更新对应元素的 SVG 属性或 `transform`；
6. 用户导出修改后的 SVG 文件。

## 场景 C：Codex 精确修改页面

1. Codex 读取项目中的结构化页面数据或源代码；
2. Codex 根据稳定的元素 ID 定位目标元素；
3. Codex 只修改指定元素；
4. 编辑器重新加载或热更新；
5. 用户继续通过可视化界面调整结果。

# 六、核心设计原则

## 1. 源代码必须是第一等公民

本工具不能只是将页面截图后进行表面编辑。

所有元素都应尽可能对应真实的：

* DOM 节点；
* CSS 样式；
* SVG 节点；
* 结构化元素对象。

用户的操作必须真实反映到底层代码中。

## 2. 双向同步

至少需要实现：

* 画布操作 → 更新代码；
* 代码修改 → 更新画布。

代码视图与画布视图不能长期处于不一致状态。

## 3. 稳定的元素标识

为了让 Codex 可以精确修改元素，需要为可编辑元素提供稳定 ID，例如：

```html
<div data-editor-id="element-001">...</div>
```

或：

```xml
<rect data-editor-id="element-002" ... />
```

不要依赖脆弱的 DOM 层级路径作为唯一定位方式。

## 4. 非破坏性局部修改

修改一个元素时，尽量不要重写整个文档，也不要无意义地格式化全部代码。

优先只修改：

* 对应节点；
* 对应属性；
* 对应样式声明；
* 与该操作直接相关的结构。

## 5. 保留原始内容

导入文件后，应尽可能保留：

* 原有 DOM 结构；
* class 名称；
* id；
* CSS 变量；
* HTML 语义；
* SVG 分组结构；
* 外部资源引用；
* 未被编辑的样式。

## 6. 对 AI 友好

页面结构应方便 GPT 或 Codex 理解和修改。

需要避免生成：

* 大量无意义的嵌套层级；
* 不稳定的随机类名；
* 难以解释的二进制状态；
* 完全依赖不可读内部格式的项目文件。

# 七、MVP 功能需求

## 1. 项目和文件导入

支持：

* 导入 `.html`；
* 导入 `.svg`；
* 直接粘贴 HTML 或 SVG 代码；
* 导入 HTML 所依赖的本地 CSS 和图片资源；
* 加载一个简单的本地项目目录；
* 提供若干示例文件用于测试。

导入后，需要对内容进行安全解析和渲染。

---

## 2. 画布

需要有一个中央画布区域，用于显示当前 HTML 页面或 SVG。

画布至少需要支持：

* 缩放；
* 平移；
* 适应窗口；
* 显示页面边界；
* 显示选中元素边界；
* 显示缩放控制点；
* 支持鼠标点击选择；
* 支持层级较深元素的选择。

HTML 页面可以优先采用固定尺寸画布，例如：

* 1920 × 1080；
* 16:9 Slide；
* 用户自定义宽高。

---

## 3. 元素选择

用户点击页面元素后，应能够：

* 选中元素；
* 显示元素类型；
* 显示稳定的元素 ID；
* 显示父元素和子元素关系；
* 显示对应 DOM 或 SVG 节点；
* 在画布、图层面板和代码视图之间同步高亮。

需要考虑以下问题：

* 嵌套元素的选择；
* 被其他元素覆盖时的选择；
* 文本节点的选择；
* SVG 分组与子元素的选择；
* 锁定元素不能被误操作。

可以提供循环选择或按层级向上选择的能力。

---

## 4. 元素变换

用户应能够直接在画布中：

* 拖动元素；
* 修改宽度和高度；
* 等比例缩放；
* 旋转元素；
* 设置精确的 X、Y、宽度、高度和角度；
* 使用键盘方向键微调位置；
* 按住修饰键进行较大步长移动；
* 锁定宽高比；
* 锁定或解锁元素。

对于 HTML 元素，需要合理处理：

* `position`；
* `top`、`left`；
* `width`、`height`；
* `transform`；
* Flexbox/Grid 中的元素；
* 百分比、像素等不同单位。

对于复杂布局，优先保证修改结果可预测。必要时，可以提供“转换为自由定位元素”的显式操作，但不要在用户无感知的情况下破坏原布局。

---

## 5. 文本编辑

支持：

* 双击直接编辑文字；
* 在属性面板中修改文字；
* 修改字体；
* 修改字号；
* 修改字重；
* 修改行高；
* 修改字间距；
* 修改文字颜色；
* 修改对齐方式；
* 修改文本框宽高；
* 支持多行文本。

编辑文字时，应尽量保留原有 HTML 标签和 SVG 文本结构。

---

## 6. 图片编辑

对于 HTML `img` 和 SVG `image`，支持：

* 替换图片；
* 修改图片路径；
* 修改尺寸；
* 修改位置；
* 修改透明度；
* 修改圆角；
* 设置 `object-fit`；
* 保持宽高比；
* 删除图片。

暂时不需要实现图片内部对象识别或位图分层。

---

## 7. 图形和样式编辑

对于 HTML 元素和基础 SVG 图形，支持修改：

* 填充颜色；
* 背景颜色；
* 描边颜色；
* 描边宽度；
* 透明度；
* 圆角；
* 阴影；
* 可见性；
* 基础滤镜；
* CSS class；
* inline style。

颜色属性应提供：

* 颜色选择器；
* HEX、RGB、RGBA 输入；
* 透明度设置。

---

## 8. 图层和结构面板

需要提供类似设计工具或 PPT 选择窗格的结构面板。

至少支持：

* 以树形结构显示 DOM/SVG 层级；
* 点击图层选中元素；
* 重命名元素的编辑器显示名称；
* 显示或隐藏元素；
* 锁定或解锁元素；
* 调整同级元素顺序；
* 删除元素；
* 复制元素；
* 查看父子关系。

HTML 和 SVG 可以共用统一的元素树抽象，但要保留各自的真实节点类型。

---

## 9. 对齐和分布

支持基础的：

* 左对齐；
* 水平居中；
* 右对齐；
* 顶部对齐；
* 垂直居中；
* 底部对齐；
* 水平等间距；
* 垂直等间距。

可以将多选能力安排在 MVP 后半阶段，但架构上需要支持多选。

---

## 10. 撤销和重做

所有用户编辑操作都应进入历史记录，包括：

* 移动；
* 缩放；
* 旋转；
* 文本修改；
* 样式修改；
* 添加和删除元素；
* 图层顺序调整；
* 图片替换。

需要支持：

* Undo；
* Redo；
* 合并连续拖动产生的大量中间状态，避免每个像素都成为一条历史记录。

---

## 11. 代码视图

提供 HTML、CSS 或 SVG 源代码视图。

需要支持：

* 语法高亮；
* 格式化；
* 搜索；
* 根据选中元素定位代码；
* 修改代码后重新渲染；
* 显示代码解析错误；
* 发生错误时不丢失上一个有效版本；
* 尽量保留代码结构和注释。

可以使用成熟的代码编辑器组件，例如 Monaco Editor 或 CodeMirror。

---

## 12. 导出

支持导出：

* 修改后的 HTML；HTML 导出应是单一、自描述且可逆的文件，可直接渲染播放，也可重新导入继续编辑；
* 修改后的 SVG；
* HTML、CSS 和资源组成的项目目录；
* 编辑器内部项目文件。

内部项目文件应包含：

* 源代码；
* 资源引用；
* 画布尺寸；
* 稳定元素 ID；
* 必要的编辑器元数据；
* 项目版本号。

项目文件不要替代标准 HTML/SVG 输出。标准格式必须始终可以导出。

对于 HTML，产品不应把“可编辑源文件”和“可播放 Slides”拆成两个互不兼容的导出结果。统一 HTML 必须保存版本化的规范文档载荷，播放器和导入器读取同一载荷；重新导入后应恢复页面、Build、稳定 ID、画布和资源语义，且不得执行导入页面原有脚本。

# 八、Codex 控制接口

这是本项目区别于普通可视化编辑器的重要能力。

请设计一个简单、清晰、可扩展的程序化控制接口，让 Codex 可以精确控制页面。

MVP 可以优先采用以下一种或多种形式：

* 本地 CLI；
* JSON Patch 文件；
* 本地 HTTP API；
* 可直接导入的 TypeScript API；
* 基于命令对象的操作接口。

建议使用结构化操作命令，例如：

```json
{
  "action": "updateElement",
  "elementId": "element-001",
  "changes": {
    "x": 320,
    "y": 180,
    "width": 640,
    "fontSize": 36,
    "color": "#202020"
  }
}
```

其他操作示例：

```json
{
  "action": "replaceText",
  "elementId": "title-001",
  "text": "New Slide Title"
}
```

```json
{
  "action": "moveElement",
  "elementId": "image-003",
  "x": 980,
  "y": 240
}
```

```json
{
  "action": "deleteElement",
  "elementId": "icon-004"
}
```

```json
{
  "action": "addElement",
  "parentId": "slide-001",
  "element": {
    "type": "text",
    "text": "New annotation",
    "x": 120,
    "y": 760
  }
}
```

程序化接口至少需要支持：

* 查询所有元素；
* 查询单个元素；
* 获取元素属性；
* 修改元素；
* 添加元素；
* 删除元素；
* 移动元素；
* 修改文本；
* 修改样式；
* 导出结果；
* 保存项目。

同时提供一个便于 AI 理解的页面结构摘要，例如：

```json
{
  "documentType": "html-slide",
  "canvas": {
    "width": 1920,
    "height": 1080
  },
  "elements": [
    {
      "id": "title-001",
      "type": "text",
      "text": "Energy-Proportional LLM Inference",
      "bounds": {
        "x": 120,
        "y": 80,
        "width": 900,
        "height": 120
      },
      "parentId": "slide-001",
      "locked": false,
      "visible": true
    }
  ]
}
```

# 九、建议的技术路线

请先评估技术方案，再开始实现。

优先考虑使用 TypeScript，并将项目拆分为清晰模块。

可以考虑以下技术栈：

* 前端框架：React + TypeScript；
* 构建工具：Vite；
* 状态管理：Zustand、Redux Toolkit 或其他轻量方案；
* 代码编辑器：Monaco Editor 或 CodeMirror；
* 元素变换：Moveable、Interact.js、Konva 或自行实现；
* HTML 渲染：受控 iframe、Shadow DOM 或隔离容器；
* SVG 渲染：原生 SVG DOM；
* 解析工具：DOMParser、PostCSS、CSSTree、Babel parser 等；
* 历史记录：命令模式或可逆 Patch；
* 本地服务：Node.js；
* 测试：Vitest + Playwright。

不要仅因为某个库容易展示效果，就将所有 HTML 内容转换成 Canvas 位图或 Fabric.js 对象。

核心目标是保留真实 DOM、CSS 和 SVG 的可编辑性。

如果使用 Canvas 库，也只能将其作为交互辅助层，不应让 Canvas 内部状态成为唯一数据源。

# 十、建议的内部架构

建议将系统划分为以下模块：

## 1. Document Model

负责统一描述：

* HTML 文档；
* SVG 文档；
* 页面；
* 元素；
* 属性；
* 样式；
* 层级；
* 稳定 ID；
* 资源。

## 2. Parser / Serializer

负责：

* HTML → Document Model；
* SVG → Document Model；
* Document Model → HTML；
* Document Model → SVG；
* 尽可能保留原始代码结构。

## 3. Renderer

负责：

* 在画布中显示 HTML；
* 在画布中显示 SVG；
* 同步元素位置和样式；
* 将元素 ID 映射到真实节点。

## 4. Interaction Layer

负责：

* 选择；
* 拖动；
* 缩放；
* 旋转；
* 多选；
* 对齐；
* 辅助线；
* 键盘操作。

## 5. Inspector

负责：

* 读取选中元素属性；
* 修改布局；
* 修改文本；
* 修改样式；
* 修改资源。

## 6. History System

负责：

* Undo；
* Redo；
* 操作合并；
* Patch 记录；
* 状态恢复。

## 7. Code Synchronization

负责：

* 画布修改同步到代码；
* 代码修改同步到画布；
* 代码错误处理；
* 防止循环更新。

## 8. AI / Codex Control Layer

负责：

* 元素查询；
* 命令执行；
* JSON 操作；
* CLI 或本地 API；
* 页面结构摘要；
* 精确修改。

---

# 十一、安全和隔离要求

由于用户可能导入 AI 生成或来源不明的 HTML，需要考虑安全问题。

至少需要：

* 默认禁止导入页面直接访问父窗口；
* 隔离脚本执行；
* 对 iframe 使用适当的 sandbox；
* 限制危险协议；
* 防止导入代码直接执行任意本地命令；
* 对外部资源加载给出提示；
* 对脚本标签采用禁用、移除或显式授权策略；
* 对事件处理器和动态代码进行限制。

MVP 可以不支持导入页面中的 JavaScript 交互逻辑。

优先支持静态 HTML、CSS 和 SVG 视觉内容。

# 新增核心能力：视觉片段、组件保存与复用

## 1. 视觉片段模型

编辑器需要支持将单个元素或多个元素组成的元素组保存为独立的 Visual Fragment。

Visual Fragment 分为：

* `element`：单个可编辑元素；
* `group`：多个元素组成的普通元素组；
* `component`：具有可编辑属性和复用能力的组件；
* `template`：具有内容插槽的可复用模板。

用户应能够：

* 保存当前选中元素；
* 保存当前选中的元素组；
* 为视觉片段命名；
* 添加描述、标签和分类；
* 生成缩略图；
* 保存到本地组件库；
* 导出视觉片段；
* 将视觉片段重新导入当前或其他项目。

## 2. Visual Fragment Package

定义一种名为 `.vfrag` 的视觉片段包格式。

`.vfrag` 本质上为 ZIP 文件，至少可以包含：

```text
fragment.vfrag
├── manifest.json
├── content.html、content.svg、content.png 或 content.jpg
├── styles.css
├── tokens.json
├── assets/
└── preview.svg
```

`manifest.json` 至少描述：

* 格式版本；
* 片段 ID；
* 名称和描述；
* 片段类型；
* 内容类型；
* 入口文件；
* 画布尺寸；
* 局部坐标系；
* 插入锚点；
* 可编辑属性；
* 内容插槽；
* 资源依赖；
* 字体依赖；
* 脚本和网络权限；
* 创建来源；
* 版本信息。

为 `manifest.json` 提供 JSON Schema，并在导入时进行格式验证。

## 3. 自包含导出

保存视觉片段时，需要自动提取其运行和渲染所需的依赖，包括：

* 匹配的 CSS 规则；
* 继承样式；
* CSS 变量；
* 字体；
* 图片；
* SVG defs；
* 渐变；
* mask；
* clipPath；
* filter；
* symbol；
* 外部资源引用。

需要提供两种保存模式：

1. Source-preserving：优先保留原始 class、CSS 和代码结构；
2. Self-contained：将视觉片段所需的样式和资源打包，使其能够在其他项目独立渲染。

## 4. 局部坐标转换

保存视觉片段时：

1. 计算选中元素组的包围盒；
2. 建立独立的局部坐标系；
3. 将内部元素坐标转换为相对于片段原点的坐标；
4. 保存原始页面位置；
5. 导入时允许放置到鼠标位置、画布中心或指定坐标。

## 5. 导入冲突处理

导入视觉片段时，需要：

* 检测重复 ID；
* 为新实例生成唯一 ID；
* 更新 HTML 属性引用；
* 更新 CSS 选择器；
* 更新 SVG href；
* 更新 `url(#id)`；
* 修复资源路径；
* 检测字体缺失；
* 检测 CSS 冲突；
* 显示导入兼容性报告。

不要因为 ID 或样式冲突而静默破坏现有页面。

## 6. 组件属性

用户可以将组件内部的部分内容暴露为组件属性，例如：

* 文本；
* 数字；
* 颜色；
* 图片；
* 图标；
* 布尔开关；
* 枚举选项；
* 尺寸；
* URL。

Codex 应能够通过组件属性修改内容，而不必直接操作内部 DOM。

示例：

```json
{
  "action": "updateComponentProperties",
  "elementId": "contribution-card-002",
  "properties": {
    "title": "Adaptive Sampling",
    "accentColor": "#315EFB"
  }
}
```

## 7. 组件插槽

组件可以定义内容插槽。

插槽需要描述：

* 插槽名称；
* 允许的元素类型；
* 是否必填；
* 是否允许多个元素；
* 默认内容；
* 尺寸限制。

Codex 和用户均可以向插槽中插入内容。

## 8. 组件实例

视觉组件导入页面后，可以选择：

* 创建独立副本；
* 创建关联实例；
* 解除实例关联；
* 更新组件定义；
* 将组件更新同步到全部实例；
* 保留实例级属性覆盖。

组件定义和组件实例必须具有不同的 ID。

## 9. 本地视觉片段库

提供本地组件库，支持：

* 缩略图；
* 搜索；
* 标签；
* 分类；
* 收藏；
* 最近使用；
* 版本；
* 来源项目；
* 使用次数；
* 导入和导出。

本地库采用 local-first：

* 用户明确选择的本地目录中的 `.vfrag` 文件是长期事实源；
* IndexedDB 只作为“临时片段剪贴板”，用于短期复制粘贴和旧数据迁移；它可能被浏览器清理，不是长期存储；
* 目录索引必须可以从 `.vfrag` 的 manifest 重建；
* 支持把临时剪贴板中的片段复制到目录，但不得自动删除原记录；
* 未连接目录时，保存默认下载 `.vfrag`；连接目录后默认直接写入目录；保存 UI 不提供 IndexedDB 目标；
* 非文字编辑状态下，`Ctrl/Cmd+C` 将画布选区写入临时片段剪贴板，`Ctrl/Cmd+V` 插入其中最新记录；HTML 粘贴根必须建立独立 stacking context 并位于当前页面最前方，组内子级保持原始层次；输入框和代码编辑器保持系统原生复制粘贴；
* 本阶段不实现账号、云端存储或同步。

允许直接导入 `.vfrag`、原始 SVG、PNG 和 JPEG。原始 SVG 保留节点树；PNG/JPEG 统一封装为 Raster element，插入后只有一个图片根图层。

Codex 控制接口需要支持查询、检查和插入视觉片段。

## 10. 多格式导出

视觉片段需要支持：

* 导出 `.vfrag`；
* 导出标准 SVG；
* 导出 HTML 和 CSS；
* 复制为 HTML；
* 复制为 SVG；
* 导出预览 PNG 或 SVG。
* 导出 Raster 的原始 PNG 或 JPEG。

自有格式不能替代标准格式。

## 11. 往返保真验收

增加以下验收要求：

1. 导入后不进行编辑，重新导出时，不应产生无意义的大规模代码变化；
2. 修改单个元素时，代码 Diff 应主要局限于目标节点及相关样式；
3. 未修改元素应尽可能保持原始结构；
4. 导出的视觉效果应与编辑器画布保持一致；
5. 导出的 `.vfrag` 在空白项目中导入后，应能够独立、正确渲染；
6. 导入包含重复 ID 的两个视觉片段后，不应出现 ID 或 SVG 引用冲突。
7. 重新连接本地目录后，应能仅根据 `.vfrag` 文件重建可用库；
8. PNG/JPEG 导入 HTML 或 SVG 后应分别成为单个 `img`/`image` 图层，且项目不得依赖原始导入文件继续存在。
